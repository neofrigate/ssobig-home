# 🚀 쏘빅 홈페이지 작업 가이드 (with Cursor)

이 문서는 비개발자 팀원들이 **Cursor(AI)**와 함께 홈페이지를 수정하고 업데이트하는 방법을 설명합니다.  
단순히 작업을 수행하는 것을 넘어, **왜 이렇게 해야 하는지** 이해하고 협업할 수 있도록 구성했습니다.

---

## 1️⃣ 작업 시작하기 (준비 단계)

작업을 시작하기 전, 다른 사람의 작업 내용을 받아오고 나만의 작업 공간을 만듭니다.

### 1. 최신 내용 가져오기

가장 먼저, 팀원들이 작업한 최신 코드를 내 컴퓨터로 가져옵니다.

> **🗣️ 프롬프트:**
> "지금 메인 브랜치의 최신 내용을 가져와서 동기화해줘"

- **📝 정식 명칭:** `git pull origin main` (풀)
- **🧐 왜 해야 하나요?**
  내가 쉬는 동안 다른 팀원이 수정한 내용이 있을 수 있습니다. 옛날 버전을 가지고 작업하면 나중에 합칠 때 문제가 생깁니다. 항상 최신 상태에서 시작해야 안전합니다.

### 2. 브랜치(작업 공간) 만들기

내 작업이 다른 사람에게 영향을 주지 않도록 나만의 공간(브랜치)을 만듭니다.

> **🗣️ 프롬프트:**
> "새로운 브랜치 만들어줘. 이름은 **[작업내용-이니셜]**로 해줘"
>
> _예시: "새로운 브랜치 만들어줘. 이름은 playroom-update-surin으로 해줘"_

- **📝 정식 명칭:** `git branch` / `git checkout` (브랜치 생성 및 이동)
- **🧐 왜 해야 하나요?**
  `main`은 실제 고객이 보는 '완성본'입니다. 여기서 바로 수정하다가 실수하면 홈페이지가 먹통이 될 수 있습니다. 복사본(브랜치)에서 안전하게 작업하고, 완성되면 합치는 것이 원칙입니다.

### 3. 개발 서버 켜기 (모바일 미리보기 포함)

작업한 내용을 실시간으로 확인하기 위해 서버를 켭니다. 폰에서도 볼 수 있게 해달라고 하세요.

**방법 A: 프롬프트로 요청하기**

> **🗣️ 프롬프트:**
> "개발 서버 켜주고, 폰에서 볼 수 있는 외부 링크도 알려줘"

**방법 B: 직접 실행하기 (더 빠름)**
터미널에 아래 명령어를 입력해도 됩니다. (이미 만들어둔 간편 실행 도구입니다)

```bash
./runserver.sh
```

> _결과 예시: https://undepressive-makenzie-supernaturally.ngrok-free.dev_ > _(내 컴퓨터에서는 http://localhost:3000 으로 접속하면 더 빠릅니다)_

- **📝 정식 명칭:** `local server` / `localhost` (로컬 서버 실행)
- **🧐 왜 해야 하나요?**
  코드를 수정했을 때 화면이 어떻게 바뀌는지 바로바로 확인해야 합니다. 특히 PC뿐만 아니라 모바일에서도 잘 보이는지 확인하는 것이 중요합니다.

---

## 2️⃣ 작업 진행하기 (수정 단계)

이제 원하는 내용을 수정합니다. 구체적으로 말할수록 정확하게 수정해줍니다.

### 1. 수정 요청하기

> **🗣️ 프롬프트 예시:**
>
> - "플레이룸 페이지의 히어로 섹션 이미지를 이걸로 바꿔줘"
> - "헤더의 폰트 사이즈를 조금 더 키워줘"
> - "하단 푸터에 인스타그램 링크를 연결해줘"

- **📝 정식 명칭:** `Coding` / `Refactoring` (코드 작성 및 수정)
- **🧐 왜 해야 하나요?**
  우리가 원하는 기능을 구현하거나 디자인을 변경하는 실제 작업 단계입니다.

### 2. 에러 해결하기 (빨간 줄이 뜰 때)

작업하다가 화면 아래 **Problems** 탭에 빨간색 숫자나 에러 메시지가 뜨면?

1. **Problems** 탭의 에러 내용을 복사합니다.
2. 채팅창에 붙여넣고 해결해달라고 합니다.
   > **🗣️ 프롬프트:**
   > "지금 이런 에러가 뜨는데 해결해줘"
   > _(복사한 에러 내용 붙여넣기)_

- **📝 정식 명칭:** `Debugging` / `Linting` (버그 수정 및 코드 검사)
- **🧐 왜 해야 하나요?**
  문법 오류가 있으면 페이지가 작동하지 않거나 배포가 되지 않습니다. 그때그때 해결해야 나중에 큰 문제로 번지지 않습니다.

---

## 3️⃣ 작업 마무리하기 (저장 및 제출)

작업이 끝났으면 내 내용을 저장하고 팀에게 알립니다.

### 1. 충돌 확인하기 (중요!)

내가 작업하는 동안 다른 사람이 메인 내용을 바꿨을 수도 있습니다. 확인이 필요합니다.

> **🗣️ 프롬프트:**
> "메인 브랜치의 최신 내용을 다시 가져와서, 지금 내 작업이랑 충돌하는 게 없는지 확인해줘"

- **📝 정식 명칭:** `Merge Conflict Check` (충돌 확인 및 해결)
- **🧐 왜 해야 하나요?**
  내가 'A 파일'을 고치고 있는데, 다른 팀원도 'A 파일'을 고쳐서 먼저 올렸을 수 있습니다. 그냥 올리면 덮어씌워지거나 에러가 납니다. 내 작업물에 남의 최신 작업을 미리 섞어서 문제가 없는지 확인해야 합니다.

### 2. 저장하고 제출하기 (PR 생성)

작업 내용을 저장(커밋)하고 검토 요청(PR)을 보냅니다.

> **🗣️ 프롬프트:**
> "지금까지 작업한 내용으로 커밋하고 PR 올려줘. PR 내용은 **[작업한 내용 요약]**으로 작성해줘"

- **📝 정식 명칭:** `Commit` (저장), `Push` (업로드), `Pull Request` (검토 요청)
- **🧐 왜 해야 하나요?**
  - **Commit**: 작업의 '저장 지점'을 만듭니다. 나중에 문제가 생기면 이 시점으로 되돌릴 수 있습니다.
  - **Push**: 내 컴퓨터에만 있는 코드를 GitHub(인터넷 저장소)에 올립니다.
  - **PR(Pull Request)**: "제 작업이 끝났으니 검토해주시고 메인에 합쳐주세요"라고 요청하는 것입니다. 팀원 간의 소통이자 안전장치입니다.

## 💡 꿀팁

- **중간중간 저장하기**: 작업이 하나 끝날 때마다 "지금까지 한 거 커밋해줘"라고 말하면 안전합니다.
- **이미지 변경**: 이미지를 바꿀 때는 "이 이미지를 **@폴더/파일이름**으로 바꿔줘"라고 구체적인 파일 경로를 알려주면 더 좋습니다.
- **한국어 설명**: 커서는 한국어를 잘 알아듣습니다. 편하게 대화하듯이 요청하세요.
